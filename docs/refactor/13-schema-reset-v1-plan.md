# Plan de reset a V1 canónica

## 1. Decisión y precondiciones

La única baseline soportada será
`backend/src/main/resources/db/migration/V1__canonical_schema.sql`. Las
migraciones V1-V060 del esquema descartado se eliminan del directorio Flyway
activo y permanecen disponibles en Git en el commit
`2764083a47303880b2e90dc6bcef807406b3ee32`.

Precondiciones operativas:

* esquema PostgreSQL 15 vacío;
* ninguna ruta de upgrade ni backfill;
* ninguna conexión a `localhost:5432`;
* validación exclusivamente con Testcontainers y puerto aleatorio;
* `ddl-auto=validate` en todos los perfiles que cargan JPA;
* si existen objetos en `public`, Flyway debe fallar por no estar vacío en vez
  de intentar normalizarlos con `IF NOT EXISTS` o `DROP`.

## 3. Orden de creación

La V1 crea directamente el estado final en este orden:

1. tablas independientes y catálogos;
2. seguridad y personas;
3. oferta académica e inscripciones;
4. asistencias;
5. orígenes cobrables e inventario;
6. cargos, pagos y aplicaciones;
7. ledgers de caja, crédito y stock;
8. recibos/outbox y notificaciones;
9. FKs no incluidas inline cuando el orden lo requiera;
10. uniques, checks e índices no cubiertos por PK/unique;
11. seed estructural mínimo del rol `ADMINISTRADOR`.

Todas las identidades son `BIGINT GENERATED BY DEFAULT AS IDENTITY`. No hay
secuencias manuales ni conversión dinámica de IDs.

## 4. Contratos de columnas

### Dinero y porcentaje

* importes/precios/saldos proyectados: `NUMERIC(19,2)`;
* porcentajes: `NUMERIC(7,4)`;
* checks `>= 0` para valores configurables;
* movimientos y aplicaciones: `> 0`;
* ningún tipo `REAL`, `DOUBLE PRECISION`, `FLOAT`, `MONEY`, `INTEGER` o `BIGINT`
  para dinero.

### Fechas y eventos

* negocio: `DATE`;
* eventos/auditoría: `TIMESTAMPTZ` con default `CURRENT_TIMESTAMP` sólo para el
  instante técnico;
* la fecha de negocio se entrega desde la aplicación mediante `Clock`.

### Textos

Longitudes se declaran igual en SQL y JPA. Observaciones y errores seguros usan
`TEXT`; usernames, estados, claves idempotentes y descripciones acotadas usan
`VARCHAR(n)`. Contraseñas se almacenan sólo como hash.

## 5. Constraints de negocio

La V1 debe incluir como mínimo:

* username normalizado único;
* una inscripción activa por alumno/disciplina (índice unique parcial);
* un horario por disciplina/día/hora;
* una mensualidad por inscripción/año/mes;
* una matrícula por alumno/año;
* una asistencia mensual por disciplina/año/mes;
* una asistencia de alumno por cabecera/inscripción;
* una asistencia diaria por registro mensual/fecha;
* un cargo por mensualidad, matrícula o venta de stock;
* `CHECK` de exactamente un origen de Cargo y coherencia con `tipo`;
* una idempotency key única por pago;
* una aplicación por pago/cargo;
* importes de pago/aplicación/movimiento positivos;
* estados válidos mediante `CHECK`;
* una reversión por pago y no doble reversión de aplicaciones;
* origen coherente y único en cada movimiento;
* cantidad de stock no negativa;
* un recibo por pago y un efecto pendiente por pago/tipo;
* `version >= 0` donde exista optimistic locking.

Las invariantes de suma (`aplicado <= cargo`, `aplicado <= pago`, crédito y stock
disponible) se protegen en un caso de uso transaccional con locks de filas y se
prueban concurrentemente. No se implementan triggers complejos como segunda
lógica de negocio.

## 6. Política de claves foráneas

* Toda FK histórica usa `ON DELETE RESTRICT` (o el default equivalente explícito).
* Toda FK usada para join, filtro o validación posee índice salvo que un unique o
  índice compuesto con la FK como prefijo ya la cubra.
* Sólo `disciplina_horarios.disciplina_id` puede usar `ON DELETE CASCADE`, por ser
  composición A demostrada.
* No hay `ON DELETE CASCADE` para inscripciones, asistencias, mensualidades,
  matrículas, cargos, pagos, aplicaciones, ledgers, egresos o recibos.
* No se usa `SET NULL` para esconder borrados; los agregados referidos se
  desactivan.

## 7. Índices previstos y consulta que justifican

| Índice | Consulta principal |
|---|---|
| alumnos `(activo, apellido, nombre)` | listado/búsqueda de alumnos activos |
| inscripciones `(alumno_id, estado)` y `(disciplina_id, estado)` | activas por alumno/disciplina |
| mensualidades `(inscripcion_id, anio, mes)` unique | generación idempotente y consulta por período |
| cargos `(alumno_id, estado, fecha_vencimiento)` | cobranza/vencidos del alumno |
| aplicaciones `(cargo_id, estado)` | saldo aplicado por cargo |
| aplicaciones `(pago_id, estado)` | recibo y monto utilizado del pago |
| pagos `(alumno_id, fecha DESC)` | historial/último pago |
| pagos `idempotency_key` unique | registro idempotente |
| movimientos_caja `(fecha, metodo_pago_id)` | caja diaria/mensual |
| movimientos_credito `(alumno_id, created_at)` | saldo e historial de crédito |
| movimientos_stock `(stock_id, created_at)` | reconciliación de existencia |
| recibos `pago_id` unique | recuperación por pago |
| recibos_pendientes `(estado, next_attempt_at)` | worker de reintentos |
| usuarios `lower(nombre_usuario)` unique | login normalizado |

No se duplica un índice ya cubierto por PK o unique. El gate de performance
validará las consultas críticas con dataset sintético determinista y
`EXPLAIN (ANALYZE, BUFFERS)`.

## 8. Implementación incremental y gates

### B1. Reset físico de migraciones

1. retirar V1-V060 del directorio activo;
2. crear V1 completa en una sola edición coherente;
3. agregar test de una migración/versión 1/catálogo sin objetos heredados;
4. ejecutar Flyway sobre PostgreSQL 15 limpio.

Gate: Flyway migrate/validate y test de catálogo verdes. Si falla, no se adapta
servicio o frontend.

### B2. Alineación JPA

1. adaptar entidades conservadas;
2. introducir sólo entidades requeridas por las tablas nuevas;
3. quitar `@Data`, cascadas y orphan removal históricos;
4. ejecutar Hibernate `validate` contra la V1.

Gate: cero discrepancias SQL/JPA y tests de relaciones verdes.

### C. Finanzas

Implementar Cargo, Pago, AplicacionPago y ledgers con transacciones, locks e
idempotencia. Primero tests públicos que reproduzcan pago parcial,
sobreaplicación, carrera y reversión; después retirar clones y resolvers.

Gate: unitarios, repositorios e integración concurrente verdes.

### D-F. Consumidores y operación

Adaptar servicios/controladores/frontend en un corte vertical que nunca deje
contratos incompatibles. Luego side effects, schedulers, seguridad, Docker/CI,
auditorías y documentación.

## 9. Verificación PostgreSQL

El test enfocado de catálogo consulta `information_schema` y `pg_catalog` y
falla si:

* hay más de una migración o la versión no es 1;
* una PK no es BIGINT identity;
* una columna monetaria no es NUMERIC(19,2);
* falta un índice de FK;
* existe un cascade no autorizado;
* faltan checks de estado;
* faltan uniques de negocio;
* existen `detalle_pagos`, `procesos_ejecutados`, `es_clon` u objetos heredados;
* Hibernate no valida el mismo catálogo.

Las auditorías SQL de `docs/refactor/sql` se reescriben para producir
`rule_id`, `entity_id` y detalle seguro sin modificar datos.

## 10. Recuperación y rollback

No existe rollback de datos anteriores porque no existe upgrade. La recuperación
consiste en:

1. conservar el commit baseline como referencia;
2. corregir V1 antes de publicar una release soportada;
3. descartar y recrear bases locales durante el desarrollo;
4. una vez publicada la nueva V1, aplicar las reglas normales de migraciones
   forward-only a cambios posteriores.

No se edita V1 después de declarar estable una instalación real. Esta
autorización sólo cubre la reconstrucción previa a producción descrita aquí.
